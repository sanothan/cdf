UPGRADE TASK: Village Management Third-Party API Feature (Reverse Geocoding)

We already use Leaflet + OpenStreetMap tiles for selecting village locations.
Now we must add an actual third-party API integration that is clearly demonstrable and aligns with rubric:
Use OpenStreetMap Nominatim Reverse Geocoding to convert lat/lng into a human-readable address and auto-fill form fields when adding a village.

GOALS:
1) Implement a backend endpoint that calls the Nominatim Reverse Geocoding API.
2) Integrate it into the Admin "Add Village" page so that clicking the map:
   - sets marker lat/lng
   - calls the backend reverse-geocode endpoint
   - auto-fills district / address fields (or shows formatted address)
3) Ensure validation, error handling, and clean architecture.
4) Update Swagger/Postman documentation for the new endpoint.
5) Make sure this endpoint is protected (admin only) OR at least authenticated, consistent with the project security approach.

---------------------------------------------
BACKEND CHANGES
---------------------------------------------

A) Create a new controller function in the Village component:
- File: backend/src/controllers/village.controller.js (or equivalent)
- Function: reverseGeocode(req, res, next)

B) Create a service function:
- File: backend/src/services/village.service.js (or create it if missing)
- Function: reverseGeocode(lat, lng)

Service should:
- Validate lat/lng are numbers and within valid range.
- Call Nominatim:
  GET https://nominatim.openstreetmap.org/reverse?format=jsonv2&lat=<lat>&lon=<lng>
- Set request headers:
  - "User-Agent": "SE3040-SanitationApp/1.0 (dev)"
  - "Accept": "application/json"
- Return a simplified response:
  {
    formattedAddress,
    district,
    state,
    country,
    raw: <optional limited fields>
  }

C) Add a new route:
- File: backend/src/routes/village.routes.js
- Route: GET /api/villages/reverse-geocode?lat=...&lng=...
- Apply auth middleware (protect) and role middleware (restrictTo('admin')) if present.

D) Add validation & errors:
- If lat/lng missing => 400
- If fetch fails => 502 Bad Gateway
- If no address found => 404 or 200 with null fields (choose consistent approach)

E) Add Swagger doc entry for this endpoint:
- Tag: Villages
- Summary: Reverse geocode coordinates
- Query params: lat, lng
- Security: BearerAuth (if protected)

Note: use native fetch (Node 18+) if available; otherwise add axios/node-fetch accordingly, but prefer minimal dependencies.

A) Locate the existing map picker component used for selecting coordinates:
- Likely: LocationPicker.jsx / MapPicker.jsx / components/Map/...
- Ensure it supports onLocationSelect({lat, lng}) callback.

B) In Admin Add Village page (AddVillage.jsx or CreateVillage.jsx):
- Maintain a form state including:
  name, district, population, waterSourceType, sanitationStatus, latitude, longitude, address
- On map click:
  1. set latitude/longitude in state

     GET {{baseUrl}}/api/villages/reverse-geocode?lat=<lat>&lng=<lng>
  3. show loading indicator "Fetching address..."
  4. auto-fill:
     - address (formattedAddress)

  5. handle error gracefully: show toast/inline message but keep coordinates

C) Ensure axios instance already includes JWT:
- Use existing axiosInstance with Authorization header
- Add a function in frontend/src/api/villages.js (if you have api layer) like:
  reverseGeocode(lat, lng)

D) UI:
- Add read-only fields for latitude/longitude (or small text lines)
- Add a read-only "Detected Address" field (editable optional)
- Keep it professional.
